我和JAVA

我雖然算不上是老程序員，但我使用JAVA 也有一段的時間了，這段期間，有不少的話想寫下來。

 

第一次使用JAVA來寫程序，是在大一那會兒的校科協授課上。他們教我使用JAVA的基本方法，那個時候，別說面向對象了，就連面向過程的代碼我基本都沒有接觸過。所以，我只覺得，哇~，JAVA似乎好厲害的樣子。像甚麼class，什麼extends，完全搞不懂。

 

第二次接觸JAVA，是1年半之後的事情了。那個時候自己也有了C語言和C++的基礎。再次接觸JAVA，感覺JAVA只不過是一個唬弄小學生的把戲，他的表層，就是C++。從C語言轉JAVA，我的感覺就像是當年我從DOTA轉到了LOL。

 

比較JAVA和C或C++的基本程序入口：

C和C++：

int main(){...}

JAVA：

public class __CLASS_NAME__{
   public static void main(String[] args){...}
}

JAVA為什麼要給入口函數之外加上一層封裝，代碼寫多了就隱隱感覺出其中的道理。

再看看基本文本終端輸出函數：

printf(...);
System.out.print(...);

感覺JAVA似乎是比C語言難不少，就我而言，我感覺JAVA並不是一種適合初學者學習的編程語言。如果你是一名初學者，我強烈的建議你不要去學習JAVA，C++這樣子的高級語言，C語言的指針也許很難，但是，如果你真的掌握了，你就會發現JAVA，C++裡面的那些高級特性，不過是對指針的封裝。C語言如果真的掌握得好，對於學習JAVA是肯定沒有壞處的。

 

因為：

JAVA的簡便之處，不在於輸出HELLOWORLD，而是他的內存自動釋放和class的寫法。

面向對象，面向物件即OPP。他的出現，應該是從大型項目中產生的，因為使用class，程序的擴展會非常的方便。

用C++寫面向對向時，最怕自己忘記做的事情就是new了之後忘記了delete，雖然這個錯誤在大部分情況下都不會發生。但是當在寫動態內存相關的程序的時候，這是經常犯的錯誤，而且是邏輯錯誤。這種錯誤我想不管多麼厲害的語法檢測工具都不可能說100%幫你查出來。造成的結果就是內存不斷的被使用，最後內存枯竭，系統藍屏。這樣的錯誤要查起來，基本就是大海撈針。

 

自從我真有一次讓我的系統藍屏了之後，我就很少再寫過動態內存的函數了。寧可開個無比巨大的array，也不想冒這樣的險。

 

而JAVA就不一樣，你完全可以不管內存的釋放。據說JAVA虛擬機會自動幫你完成你需要做的這項工作。可以讓程序員集中精力在程序的核心部分，而不是浪費時間去找內存錯誤這樣的bug的原因。

 

然，JAVA的優點似乎也就到此為止了。

 

以前我寫JAVA，因為基本不會寫什麼大項目，就寫個HelloWorld之類的小程序，也用不著什麼IDE，用用javac就足夠了。後來，因為寫的項目越來越大，漸漸也感覺到javac有點力不從心，就接觸了eclipse。覺得還挺好用的，至少比javac好用。似乎也不比visual studio差。

但我剛把話說完，我就知道我錯了。就現在的我，我感覺，自己估計是再也不會去使用eclipse，也不再也不會去用JAVA寫程序了。因為，訴我直言，JAVA，就是一個智障，eclipse，就是智障他爹。而，用eclipse寫JAVA的我，就是智障他孫子。

註釋，自古就是程序員逃避編譯檢測的最好的手段，而你，***居然會去檢查我註釋裡面的東西，而且，居然還會給我報錯？！你***，你放一個警告我也不會怪你智障。而你，非得顯示出自己無比強大且智障的語法檢測能力，在我的註釋中橫行霸道。我*，你就是一個傻逼。

 

我就問你，註釋裡面的東西，你是不是還會幫我編譯到字節碼裡面？然後方便黑客的反編譯？是的話，我謝謝你。不是的話，你**是大腦有屎還是小腦出翔的給我去查我註釋裡面的東西？我也是服。我聽過一句話，不要試圖去戰勝一個傻逼，因為他會把你的智商拉倒和他一個水平，然後再用他豐富的經驗打敗你。我，我真的感覺，為了逃避編譯檢測而去修改註釋裡面內容的我，就是那個智商被JAVA拉低的娃，我，真的被你打敗了！

 

我特意去網上看了一下，遇到這種情況的人，還不只我一個。但是，回答的人卻告訴我，這是JAVA的標準。請遵守他。

 

算了，我已經沒有精力去給你糾結這些破事了。我還聽過一句話，傻逼活著會比較輕鬆。我就老老實實的當一個傻逼吧。不要為了JAVA的一個註釋，而累得半死。

 

那個給我報錯的語句如下：

/* copy: Fundamental-2D-Game-Programming-With-Java-master\CH11\src\javagames\util\utility.java */

報錯的地方： \u

JAVA的答覆是，\u後面必須接4個數字，因為會把他們解析成一個unicode的編碼字元。

 

其實在C和C++定的char型中也可以通過\u+四個數字進行設置，比如：

wchar_t ch = "\u1234"

但是，就算你後面沒有跟四個數字，編譯器也只會給你個警告，而不像你JAVA，他媽的寫在註釋裡都給我報錯。

if語句 中的整數

之後，第二個問題又來了

 

我們都知道，JAVA是不允許你在應該寫邏輯表達式的地方寫賦值表達式的。C和C++則會去判斷賦值表達式所設置的值是不是0。

這樣的語句是會被JAVA報錯的：

if(1){...}

這有好處，也有壞處吧。仁者見仁，智者見智。我就不在此發表我的牢騷了。不過我還想說幾句：

 

而在C語言中，我基本都是使用那種JAVA會報錯的寫法去寫邏輯表達式的。就算不小心寫錯了，我查代碼花的時間，絕對不超過10分鐘。恩，不會像查內存洩漏那樣的代碼要花10小時。在某些時候，如果能夠把他們寫在一起，代碼會漂亮很多。

 

JAVA強制要求程序員寫邏輯語句，這樣子的要求我是感覺沒有必要。骨灰級玩家從來不會遵守新手教程裡面的要求。我也從來沒有因為哪一份代碼中應該寫邏輯表達式的地方寫了賦值表達式而感到困惑過。這樣子的要求，對於代碼的可讀性，我感覺，完完全全沒有任何幫助，我也說過，就算自己不小心犯了這樣子的錯誤，我查代碼用的時間也絕不會超過10分鐘。也許，這樣的要求利於新手程序員養成良好的編程習慣吧。但看慣了步兵的娃去看騎兵的時候，會有什麼樣的心情呢？我想，這群新手程序員以後讀UNIX內核代碼的時候就會體驗到了。

 

這個要求，作為一個標準來說，是有好處的。我也儘量要求自己不要在該寫邏輯表達式的地方寫賦值表達式。但是，這做為一個強制要求，真的，就很過分了。

省略return的錯誤

還有第三個問題

 

我在用visual studio寫C++代碼的時候，都會習慣性的在開後添加一句代碼：

#pragma warning(disable:4716)

這句代碼的作用是，讓所有關於應該有返回值的函數卻沒有寫return的錯誤閉嘴。

 

你問我，為什麼不給那些函數寫上return？因為，他們的返回值沒有被用上啊。我又何必浪費時間去給他們加上完全沒有意義的return 0這樣的語句呢？

你又問我，既然他們的返回值沒有被用上，那麼，為什麼不給他們設定成void型的返回值？因為，現在沒有被用上，指不定哪天就被我用上了呢？你知道重構代碼的痛苦嗎？既然知道，就在一開始把接口寫好不就行了？其實printf函數也有返回值，你用過他的返回值嗎？沒有吧？但是，指不定哪天你就用上了呢。所以，在一開始把它設定成有返回值，總是沒有錯的。

 

那麼，對於應該有返回值的地方卻沒有return語句這樣的情況，JAVA是怎麼處理的呢？

報錯，報錯，報錯，報錯

 

而且，JAVA的錯誤提示比visual studio明顯得多，vs頂多是在出錯的地方顯示一條只有一個字元長度的紅色下滑縣，然後在右邊的滾動條那裡出錯的語句所在的行顯示一個紅點。有時候我找出錯的地方都要睜大眼睛仔細的看。而eclipse卻是，一條長長的下滑縣，明顯得不能再明顯，生怕你還看不到。這讓我這種有強迫症的人非常的難受。但現在我完全沒有罵人的衝動，因為，確實，該寫return的地方沒有return是我程序員的錯。只是編譯器你有點過分了。

我*，我**的，我只是為了方便debug而寫的一個return，你***，我，我好好的一個寫程序的心情都被你這種傻逼編譯器給毀完了。你**當我是小學生啊，return後面的代碼不會被執行你當我不懂啊？你他媽這樣的東西都來提示我，你是不是閒得發慌顯用黃鱔來自慰啊？

煩人的自動完成

你以為JAVA的智障只有這麼一點？如果真的是，那麼，也許我會懷著我博大的胸懷去接受它，畢竟，他的內存釋放和class的寫法，真的比C++好用。可惜了，他的老爸eclipse，卻讓我永遠的拋棄了他。

一個機智的IDE總可以讓人心情很好，我想，所有的程序員都會有這樣的感觸。我喜歡把IDE設置成自己喜歡和習慣的樣子，這樣的話，我就會覺得非常的熟悉，寫起代碼，也會非常的有感覺。

所有的IDE，不管是eclipse還是vs，我拿過來的第一件事情，就是把代碼自動完成給關了。vc6.0除外，因為他根本就沒有這個功能。

 

大一的用vc6.0的時候，我感覺非常的辛苦，恩，對是辛苦。非常的辛苦。如果，vc6.0能夠提供自動完成功能，那麼那個時候的自己也許會覺得稍微有點輕鬆。可惜了，他沒有。那個時候，我也沒有用過vs。就這樣，在vc6.0的陪伴之下，我度過了大一一年，也用vc6.0寫出了一些自己滿意的作品。

 

後來，接觸了vs，我一次用vs時，一條語句，我輸入完畢，在最後按下;時，vs很機智的把所有應該有空格的地方，都給我補上了空格。我的感覺是，哇~這樣我的代碼就變得好漂亮了啊。不過這個感慨，沒有持續過1個月，我就把vs的自動完成給關閉了。雖然他可以讓我的代碼變得十分的漂亮，但是如果我的代碼出了問題，需要修改的時候，我肯定就不會去輸入;讓他幫我格式化。而每一次 Ctrl+A -> Ctrl+K Ctrl+F 都會用掉我大筆的寶貴時間。每次讀我改過的代碼的時候我都會感覺非常的不適應，因為那份代碼裡面有兩種不同的代碼風格。我覺得，就算我原本的代碼風格不是很好，也比一份代碼中存在兩種不同的代碼風格要好。再加上，我漸漸的習慣使用table來進行對齊，也不想讓編譯器給我操心這樣的事情了，所以，我關閉了vs中所有的自動完成。

 

在那之後，我接觸過的每一款IDE，我都會選擇關閉他的自動完成。不管是[](){}的尾括號補全，還是代碼的自動格式化。作為一個程序員，我想，敲一個符號都是秒秒鐘的事。我們每天敲的代碼不下100行，還在乎你一個尾括號的工作量？而代碼的自動格式化會破壞我們原本的代碼風格。

 

所以，拿到JAVA的時後，我也費了好大的功夫，關閉了他基本上所有的自動完成。但是，有一點，我卻怎麼都關不掉。就是，當你輸入/*，按下會車後，他會幫你在下一行的開頭加入一個*。

是智能還是智障

而對於JAVA，我還有一點很疑惑的地方。我不知道這是我太年輕了，還是JAVA真的是一個智障。

JAVA和C++一樣，有一種機制叫做繼承，但是，JAVA卻有一個邪惡的要求，Constructor call must be the first statement in a constructor。意思很簡單，就是，你如果想用你的子來初始化你的父，那麼，你不需要把初始化父的代碼寫在除了註釋以外的第一行。

舉個例子，

class father{
	father(int a){};
}
class child extends father{
	child(){
		int a;
		super(1);//構造父
	};
}

這樣的代碼是不可以通過的。因為，構造父的代碼不是子的構造函數中的第一條語句。你必須要把a的申請放倒super(1)後面。

class child extends father{
	child(){
		super(1);
		int a;
	};
}

但是，智障的地方出現了，這樣的代碼是可以編譯過的：

class child extends father{
	child(){
		super(1+2);
	};
}

你甚至可以加入一個add函數讓他編譯通過﹕

int add(int a,int b){
	return a+b;
}
class child extends father{
	child(){
		super(add(1,2));
	};
}

而這樣的代碼卻不行：

class child extends father{
	child(){
		int a = 1+2;
		super(a);
	};
}

我真的不明白為什麼，是因為在構造父之前，申請了內存會導致內存對齊的問題？我想不明白。

有前輩曾說過，不要認為你比編程語言的創造者或寫頭文件的人聰明，那怕只有一點點。所以，我感覺，JAVA設置了這樣的強制性規定，一定是有它的原因的。但是，"Constructor call must be the first statement in a constructor"這樣的理由，我不能接受。在論壇上我看過其他程序員的發言，他們說，在編譯器生成的字節碼文件裡面是沒有這樣子的規定的。難道？這又是JAVA的智障要求？
